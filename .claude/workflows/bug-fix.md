# Workflow: Bug Fix

버그를 수정하는 워크플로우.

---

## 개요

| 항목 | 내용 |
|------|------|
| **목적** | 버그 신속하게 수정 |
| **핵심 원칙** | 재현 → 원인 분석 → 수정 → 검증 |

---

## 단계별 흐름

```
Step 1: 버그 분석
    CEO/qa-engineer → planner → 담당 Agent
    
Step 2: 수정
    frontend-dev / backend-dev
    
Step 3: 검증
    qa-engineer → reviewer → planner
```

---

## Step 1: 버그 분석

### 1.1 버그 리포트 확인

**planner 작업:**
```markdown
## 버그 분석: {버그 제목}

### 버그 정보
- 심각도: {Critical / High / Medium / Low}
- 재현 단계: {재현 방법}
- 예상 동작: {예상}
- 실제 동작: {실제}

### 원인 분석
- 예상 원인: {예상 원인}
- 관련 파일: {파일 경로}
- 담당: {frontend-dev / backend-dev}
```

### 1.2 담당 Agent 할당

**심각도별 우선순위:**

| 심각도 | 대응 |
|--------|------|
| **Critical** | 즉시 수정, 다른 작업 중단 |
| **High** | 당일 수정 |
| **Medium** | 일반 우선순위 |
| **Low** | 여유 있을 때 |

---

## Step 2: 수정

### 2.1 원인 파악

**frontend-dev / backend-dev 작업:**
- 버그 재현 확인
- 코드 분석
- 원인 특정

### 2.2 수정

**frontend-dev / backend-dev 작업:**
- 버그 수정
- 사이드 이펙트 확인
- 로컬 테스트

### 2.3 수정 보고

```markdown
@planner 버그 수정 완료

## 버그: {버그 제목}

### 원인
{원인 설명}

### 수정 내용
{수정 내용}

### 수정 파일
- {파일 1}
- {파일 2}

### 테스트 방법
{확인 방법}

검증 요청드립니다.
```

---

## Step 3: 검증

### 3.1 QA 검증

**qa-engineer 작업:**
- 버그 수정 확인 (재현 단계 재실행)
- 회귀 테스트 (관련 기능)
- 결과 기록

```markdown
## 버그 검증 결과

### 수정 확인
- [ ] 원래 버그 재현 안 됨 ✅

### 회귀 테스트
- [ ] 관련 기능 정상 동작 ✅

### 결과: Pass / Fail
```

### 3.2 검수 (Critical/High만)

**reviewer 작업:** (Critical, High 심각도만)
- 수정 코드 검토
- 사이드 이펙트 확인

### 3.3 완료

**planner 작업:**
```markdown
@CEO 버그 수정 완료

## 버그: {버그 제목}
- 원인: {원인}
- 수정: {수정 내용}
- 검증: 완료

배포 진행해도 될까요?
```

---

## 체크리스트

### 분석
- [ ] 버그 재현 확인
- [ ] 원인 파악
- [ ] 영향 범위 확인

### 수정
- [ ] 버그 수정
- [ ] 사이드 이펙트 없음
- [ ] 로컬 테스트 완료

### 검증
- [ ] QA 검증 통과
- [ ] 회귀 테스트 통과
- [ ] (Critical/High) reviewer 검수 통과

---

## 버그 우선순위 가이드

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **Critical** | 서비스 불가, 데이터 손실 | 앱 크래시, 결제 오류, 로그인 불가 |
| **High** | 주요 기능 오작동 | 핵심 기능 에러, 데이터 표시 오류 |
| **Medium** | 기능 일부 문제 | UI 깨짐, 사소한 기능 오류 |
| **Low** | 경미한 이슈 | 오타, 미세한 UI 이슈 |
